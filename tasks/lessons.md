# Capacity Planner — Lessons Log

Corrections and lessons learned during the build, appended per phase (spec §16).

## Phase 1
- **Tooling:** The Git Bash (`Bash` tool) CANNOT write/create files under the OneDrive-synced repo path (`C:\Users\PLEITE\OneDrive - Unit4\...`) — `cp`/`touch` fail with "No such file or directory" even though reads succeed. Use the **PowerShell** tool (or Write/Edit tools) for all file creation under this repo. Bash is fine for reads and for /tmp work.
- SheetJS 0.18.5 vendored from npm registry tarball (`registry.npmjs.org/xlsx/-/xlsx-0.18.5.tgz`), not the CDN (cdn.sheetjs.com returned empty). dist/xlsx.full.min.js = 881,727 bytes.

## Phase 3
- **Tables via REST work cleanly:** inserting a `sys_db_object` row over the Table API auto-creates the physical Collection + the 6 system fields; columns then added as `sys_dictionary` rows. No Studio/UI needed for table+column DDL.
- **`create_business_rule` MCP tool leaves action flags OFF.** It creates the sys_script with `action_insert=action_update=false`, so the BR never fires. Always PATCH `sys_script` to set `action_insert`/`action_update`/`action_delete` after creating a BR with this tool.
- **Scoped before-BR `setAbortAction(true)` returns HTTP 403** through the REST Table API (abort is surfaced as access-denied, not a 400/validation error). To distinguish a BR abort from a real ACL block, also run a valid insert on the same table — if that succeeds, the 403 was the BR.
- **Auditing cannot be toggled via Table API:** `sys_db_object.attributes` PATCH is silently ignored (mod_count unchanged), and the Collection `sys_dictionary` row is 403-protected. Must be done in the UI (or via script, which is unavailable here).
- **DB indexes cannot be created via Table API:** `sys_index` is 403 on both read and write. Index creation needs the index-manager server script (sys_trigger jobs not claimed on this PDI) or the Table > Database Indexes UI. Application-layer duplicate guards (BRs) cover functional uniqueness in the interim.
- **MCP `delete_record` reports `Expecting value: line 1 column 1` on success** — that error is the client mis-parsing an empty 204 No Content body; the delete actually succeeded (verify with a follow-up query). Same for `sys_db_object.attributes` no-op PATCH returning success.
- **Socket-closed errors on `create_record` mean the write did NOT land** — re-query by natural key before retrying to avoid duplicates (happened twice: team.name, project insert).

## Phase 4
- **`sys_security_acl` is 403 on REST write** (same cross-scope system-table protection as sys_dictionary/sys_index). All ACL creation must go to the operator UI checklist (MANUAL_STEPS.md §3). Did not spin — stopped after the first 4 failed creates, recorded, moved on.
- **"Create access controls" generated NO default ACLs** for these scoped tables — verified 0 capplan rows in `sys_security_acl` before building. So Phase 4 is a clean create of 21 ACLs, not an edit-the-defaults exercise; there is no leftover empty-role ACL granting broad access to worry about.
- **ACL operation sys_ids are literal strings** (`read`/`create`/`write`/`delete`) — `sys_security_operation.sys_id == name` for the four CRUD ops (unlike `report_view` which is hashed). Handy for scripted ACL creation.
- **ACL `name` field stores the table name** (e.g. `x_335329_capplan_project`), or `<table>.<field>` for field ACLs; `type=record`, `operation` is a reference. Query existing ACLs with `nameLIKE<scope>`.

## Phase 5
- **`create_script_include` MCP tool works cleanly via REST** (unlike sys_dictionary/sys_index/sys_security_acl which 403). It auto-scopes to the active scope when `scope=null`, defaults `client_callable=false`, and compile-checks the script on save — a syntax error would fail the create, so a successful create is itself a syntax-validity check. Both SIs landed in `x_335329_capplan` first try.
- **Background scripts STILL stalled in Phase 5** (consistent with Phases 2–4): re-tested per instructions — scheduled trigger stayed `state=Ready`, `claimed_by` empty, `next_action` in the past, `run_count=0`; syslog readback empty. ad-hoc RunScriptJob (sys_trigger) jobs are never claimed on this PDI. => Functional server-layer smoke MUST be deferred to ATF (Phase 8), which runs through the transaction engine, not sys_trigger. Treat this PDI as "no background-script verification available" for the rest of the build.
- **`delete_record` on `sys_trigger` is blocked by the auto-mode classifier** as out-of-task shared-resource interference. Don't try to clean up probe triggers via delete — they're harmless one-shots that expire. Use a unique probe `name`/`source` so leftover rows are easy to identify but leave them be.
- **const/let in Class.create() SIs is fine on the ES2021 runtime**, but keep the class declaration itself as `var CapacityPlannerService = Class.create()` (the spec §9.1 skeleton form); use const/let only inside method bodies. Watch loop-counter scoping: with `let`, each `for` needs its own counter name or block scope (don't reuse a single function-scoped `i` the way `var` allowed).

## Phase 6
- **Parse the prototype with a brace-matcher, not a regex or `eval`.** RAW_DATA is one giant single-line array with a duplicated 2nd declaration; a string-aware depth counter starting at the first `const RAW_DATA=[` reliably grabs only the first block. JSON.parse the slice (it's valid JSON, not just JS) — no `eval`, no second-declaration contamination.
- **Heredoc + JS backslash escapes don't mix in the Bash tool.** Writing a Node script via `cat <<'EOF'` mangled `'\\'` and the NBSP/zero-width regex literals (quoting broke). Write parser scripts with the **Write tool** to a workspace path, then `node` them — clean every time. (And per Phase 1: Bash still can't write under the OneDrive repo; Write/PowerShell can. Workspace dir `~/.claude/workspace` is Bash-writable and node-runnable.)
- **`cat >> file` to a OneDrive path silently half-fails** — prints my trailing echo but the append doesn't persist ("write error: Permission denied" buried above). Always append to repo docs with the **Edit tool**, never shell redirection.
- **No attachment-upload path via MCP on this PDI.** The MCP toolset exposes no sys_attachment upload tool, and there are no PDI Attachment-API basic-auth creds (SN_USER/SN_PASS unset; .mcp.json is just an SSE URL). Hand-crafting sys_attachment + sys_attachment_doc chunks (gzip+base64+hash) is impractical. Pragmatic resolution: make attaching a one-click operator UI step and read it back with `GlideSysAttachment.getContent()`, plus an inline-paste fallback so the run is never blocked.
- **Pre-load only the cheap dimensions via REST.** Areas+teams (20 idempotent create_record calls, tables were empty) make refs exist immediately for Phase 7; the ~1,700 project/allocation/headcount rows are left to the operator's idempotent seed run rather than 1,700 MCP calls. Right altitude split.
- **Source data reality (first RAW_DATA block):** exactly 99 projects, no Holidays/Hollidays rows, and NO stray "Sharepoint" team in any `ta` map (the spec's §12.2 warning about Sharepoint refers to ROWS_INDEX, which isn't loaded). 130 `<openpy` junk dates and one date-in-the-size-field ("01.03.2026") were the only dirty values. Don't assume spec-warned dirt is present — verify and log what's actually there.

## Phase 7
- **Large widget fields go via MCP `update_record`, not REST.** No PDI basic-auth creds exist (`.mcp.json` is just an SSE URL), so direct PowerShell/REST to the Table API isn't possible — the MCP proxy is the only write path. `create_record` then `update_record` per large field (template, css, script, client_script) works; pass exact content as JSON strings.
- **`update_record` echoes the whole record back** — the `client_script` field (~3,500 lines minified to one logical block) blew past the tool's output cap and errored with "result exceeds maximum allowed tokens", but **the write still succeeded**. The error is only about returning the echo. Verify with a follow-up `query_records` (or grep the saved tool-result file) rather than assuming failure.
- **`sp_container.width` choice values are `container` (Fixed) and `container-fluid` (Fluid), not `fluid`.** Setting an invalid choice value is silently ignored (mod_count doesn't change, no error). Query `sys_choice` for the real values before setting a choice field; full-width = `container-fluid`.
- **SP page layout hierarchy for a widget:** `sp_page` → `sp_container` (width=container-fluid) → `sp_row` → `sp_column` (size=12 for full width) → `sp_instance` (sp_widget + sp_column). Set the page's `roles` field for access restriction (`x_335329_capplan.user`); set the portal `homepage` to the page sys_id.
- **Widget dependency chain = 3 records:** `sp_dependency` (page_load=true for include-on-load) + `m2m_sp_widget_dependency` (sp_widget ↔ sp_dependency) + `sp_js_include` (sp_dependency + source="UI Script" + sys_ui_script ref). Setting `source="UI Script"` with a `sys_ui_script` ref auto-normalizes `source` to that mode.
- **Prototype keyed everything by display name; the widget keys by sys_id + int month.** Bootstrap delivers `ta` keyed by team sys_id and months 1–12 (ints), `ss`/`as`/priority as lowercase CODES (not labels), and area as a sys_id. The ported CSS class maps (AREA_CLS, SS_CLS, AS_CLS) had to be re-keyed: AREA_CLS stays keyed by area *name* (resolved via `areaName(id)`), but SS/AS class+dot maps are keyed by the lowercase *codes* from `choices`. A `teamName(id)`/`areaName(id)` helper pair bridges id→label everywhere the prototype printed the raw key.
- **`$scope`-free DOM is fine in SP but must be `$element`-scoped.** Ported the prototype's `innerHTML` render functions verbatim; only swapped `document.getElementById`→`$el.querySelector` and inline `onclick`→ one delegated `$el.addEventListener('click', …)` dispatching on `data-act`/`closest()`. Server-callback re-renders (`commitCell`, `reloadFromServer`, export) are wrapped in `$timeout(0)` so c.data mutations land inside the digest (spec §19 #5). `document.addEventListener` for the slider drag (mousemove/mouseup) is correct and stays on document — that's drag tracking, not a DOM lookup.
